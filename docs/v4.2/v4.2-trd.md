# v4.2 TRD — Technical Requirements Document

**Version:** 0.1 (Draft)
**Companion to:** `v4.2-prd.md`
**Status:** Initial draft
**Date:** 2026-04-12
**Scope:** Gene balancing, visual system, and targeted mechanic improvements. Backwards compatible with v4.x.

---

## 1. Goals and Constraints

### 1.1 Goals

1. **Continuous-only trait expression** — replace v4's hard min/max clamping with soft floors and unbounded ceilings, governed by trade-off cost functions.
2. **Modular rendering additions** — universal animations, configurable indicator slots, generalized tool rendering.
3. **Geographic isolation** — terrain generation that creates separated population pockets.
4. **Pregnancy overhaul** — gradual need transfer from parent to offspring during gestation.
5. **Leveling cost scaling** — energy cost increases per level to balance the Aptitude trait.
6. **New genes** — Sociality (AD, replaces Gregariousness OO), Pregnancy (AG), Volatility (AP).
7. **Backward compatibility** — v4.x save files must load without data loss.

### 1.2 Constraints

- **Domain-driven, event-bus communication** — preserve v4's architectural principles (no circular imports, no god objects).
- **No new dependencies** — TypeScript + esbuild + Canvas 2D, same as v4.
- **No test framework yet** — but write code that's testable (pure functions, dependency injection where reasonable).
- **Strict TypeScript, no `any`** — same as v4.
- **Browser-only target** — no Node-specific APIs.
- **All `.ts` files ≤ ~300 lines** — split if longer.
- **No structural renames** — Agent remains Agent, trees remain trees. No entity model changes.

### 1.3 Non-Goals

- Unified organism model (trees as agents)
- Phenotype classification system
- Simplified tick for immobile entities
- Meat/plant food type split, corpse blocks
- New Body & Physiology traits (Size, Regeneration)
- New Mobility/Senses/Diet traits (Aquatic, Saltwater Tolerance, Photosynthesis, Carnivory)
- Functional minimum thresholds system (capabilities remain gene-driven as in v4)
- Breaking save format changes

## 2. Architecture Principles

### 2.1 Inherited from v4 (unchanged)

1. **Domain-driven.** Each domain owns its data, logic, and types. No domain reaches into another's internals.
2. **Event-driven communication.** Domains emit events through a central `EventBus`. Cross-domain side effects use events, not direct calls.
3. **Composition over inheritance.** Agents are composed of components (Genome, Needs, Inventory, Relationships, Pregnancy), not subclassed.
4. **Data flows down, events flow up.** The simulation tick reads state and calls domain update methods. Domains emit events when noteworthy things happen.
5. **Per-agent configuration.** All game-mechanical values that vary per agent come from genetics. Global constants are reserved for truly universal values (tick rate, grid size).

### 2.2 New for v4.2

6. **Continuous-only trait expression.** Every trait is a continuous value. The v4 hard min/max clamping is replaced with soft floors and no hard ceilings.
7. **Cost scaling replaces hard caps.** Trait expression has no upper bound; cost functions on actions and passive drains are the natural limiter.

## 3. Tuning Constants (`core/tuning.ts`) — NEW

All non-universal numeric tuning lives in a single `tuning.ts` module exporting `TUNE`. This separates tuning from code, enabling rapid iteration without hunting through files.

```typescript
export const TUNE = {
  // Cost scaling coefficients
  cost: {
    attackEnergyPerStrength: 0.015,        // Energy per attack = base + (strength × this)
    moveEnergyPerAgility: 0.008,
    recallSlotPassiveDrain: 0.0008,
    immunityPassiveDrain: 0.002,
    levelEnergyMultPerLevel: 0.04,          // +4% energy cost per level gained
  },

  // Pregnancy
  pregnancy: {
    needTransferRate: 0.4,                  // Per-tick fraction of need transferred
    completionThreshold: 80,                // Child needs reach this → birth
  },

  // Mutation
  mutation: {
    baseRate: 0.005,                        // At default Volatility
    geneDupChance: 0.01,
    geneDelChance: 0.01,
    minDnaLength: 100,
    maxDnaLength: 250,                      // Same as v4
  },

  // Decay rates (defaults — actual values come from genes per agent)
  decay: {
    baseFullnessPerTick: 0.03,
    baseHygienePerTick: 0.02,
    baseSocialPerTick: 0.01,
    baseEnergyPerTick: 0.01,
  },

  // World generation
  world: {
    obstacleDensity: 0.08,                  // Fraction of cells as obstacles
    isolationPocketCount: 4,                // Target number of regions
  },
} as const;

export type TuneConfig = typeof TUNE;
```

This module is imported by domains that need numerical tuning. The TUNE object is `as const` for compile-time safety.

## 4. Domain: Genetics

**Responsibility:** DNA encoding, gene parsing, trait expression, crossover, mutation, viability, cost calculation. Pure functions only — no side effects, no world state.

### 4.1 New Gene Codes

v4.2 adds three new gene codes to the existing v4 registry:

```typescript
// New in v4.2
'AD' Sociality         // Replaces OO Gregariousness
'AG' Pregnancy         // Gestation duration (new mechanic)
'AP' Volatility        // Per-lineage mutation rate
```

All other v4 gene codes (AA–VV) remain unchanged.

**Migration note:** Existing OO (Gregariousness) genes in v4 saves are treated as equivalent to AD (Sociality) during load. New agents are seeded with AD instead of OO.

### 4.2 TraitSet Changes

The v4 `TraitSet` is extended with new fields for the three new genes. Existing fields are unchanged.

```typescript
// Added to TraitSet
export interface TraitSet {
  // ... all existing v4 fields unchanged ...

  // v4.2 additions
  sociality:    { value: number };           // Replaces gregariousness
  pregnancy:    { gestationMs: number };     // 0 = instant birth
  volatility:   { mutationRate: number };    // Default = TUNE.mutation.baseRate
}
```

When expressing DNA that lacks AD, AG, or AP genes, the expression function returns default values:
- `sociality.value` → same as the agent's gregariousness expression (migration path)
- `pregnancy.gestationMs` → 0 (instant birth, matching v4 behavior)
- `volatility.mutationRate` → `TUNE.mutation.baseRate` (matching v4's global constant)

### 4.3 Expression Function (`genetics/expression.ts`)

Replaces v4's `clamp(default + (direction × raw / scale), min, max)` with `max(floor, default + (direction × raw / scale))`.

```typescript
export function expressGenome(dna: string): TraitSet {
  const genes = parseDna(dna);
  const traitSet: Partial<TraitSet> = {};

  for (const [code, def] of GENE_REGISTRY) {
    const matchingGenes = genes.filter(g => g.code.toUpperCase() === code);
    const positive = matchingGenes
      .filter(g => isReinforcing(g.code))
      .reduce((sum, g) => sum + g.magnitude, 0);
    const negative = matchingGenes
      .filter(g => !isReinforcing(g.code))
      .reduce((sum, g) => sum + g.magnitude, 0);
    const raw = positive - negative;

    for (const comp of def.components) {
      const direction = comp.inverted ? -1 : 1;
      const value = Math.max(
        comp.floor,
        comp.default + (direction * raw / comp.scale)
      );
      // NOTE: no Math.min(comp.max, ...) — v4.2 removes hard ceilings
      assignTraitComponent(traitSet, def, comp, value);
    }
  }

  return finalizeTraitSet(traitSet);
}
```

The key change from v4: **no `Math.min(comp.max, ...)` clamp.** Traits are unbounded above; cost functions limit them naturally.

### 4.4 Cost Functions (`genetics/cost-functions.ts`) — NEW

This module exposes pure functions that derive per-agent cost values from a `TraitSet`. Other domains call these instead of using global TUNE values directly when the cost should scale with traits.

```typescript
export function attackEnergyCost(traits: TraitSet): number {
  return TUNE.cost.attackEnergyBase
    + traits.strength.baseAttack * TUNE.cost.attackEnergyPerStrength;
}

export function moveEnergyCost(traits: TraitSet): number {
  return TUNE.cost.moveEnergyBase
    + traits.agility.speedMult * TUNE.cost.moveEnergyPerAgility;
}

export function passiveEnergyDrainPerTick(traits: TraitSet): number {
  return TUNE.decay.baseEnergyPerTick
    + traits.recall.memorySlots * TUNE.cost.recallSlotPassiveDrain;
}

export function levelEnergyMultiplier(level: number): number {
  return 1 + (level - 1) * TUNE.cost.levelEnergyMultPerLevel;
}
```

These are referenced by `action-factory.ts` (when materializing an action's cost) and `agent-updater.ts` (per-tick passive drain).

### 4.5 Viability (`genetics/viability.ts`)

```typescript
export function isViable(traits: TraitSet, dna: string): boolean {
  // Essential traits
  if (traits.longevity.maxAgeMs <= 0) return false;
  if (traits.metabolism.fullnessDecay <= 0) return false;

  // Energy pool and HP
  if (traits.vigor.baseMaxEnergy <= 0) return false;
  if (traits.resilience.baseMaxHp <= 0) return false;

  // DNA length
  if (dna.length < TUNE.mutation.minDnaLength) return false;
  if (dna.length > TUNE.mutation.maxDnaLength) return false;

  return true;
}
```

Note: There is **no separate stillbirth probability**. Viability is the only stillbirth check. Volatility increases stillbirth indirectly by mutating more genes, sometimes breaking essentials.

### 4.6 Mutation (`genetics/mutation.ts`)

```typescript
export function mutate(dna: string, mutationRate: number): string {
  // Same logic as v4 but `mutationRate` is now a parameter
  // (was constant 0.005 in v4)
  const chars = dna.split('');
  for (let i = 0; i < chars.length; i++) {
    if (Math.random() < mutationRate) {
      chars[i] = randomChar(i);
    }
  }
  // ... gene duplication, gene deletion, length clamp, padding (unchanged from v4)
}
```

Callers (in `reproduce-effects.ts`) compute the rate from parents' Volatility:

```typescript
// Sexual reproduction
const childMutationRate = (parentA.traits.volatility.mutationRate
                         + parentB.traits.volatility.mutationRate) / 2;
const childDna = mutate(crossover(parentA.dna, parentB.dna), childMutationRate);

// Asexual reproduction
const childMutationRate = parent.traits.volatility.mutationRate;
const childDna = mutate(parent.dna, childMutationRate);
```

## 5. Pregnancy Transfer Mechanic

### 5.1 PregnancyState (`entity/components/pregnancy.ts`)

v4.2's pregnancy tracks the offspring's needs being filled gradually:

```typescript
export class PregnancyState {
  active: boolean;
  childDna: string | null;
  childFamilyName: string | null;
  childFactionId: string | null;
  partnerId: string | null;

  // v4.2 transfer mechanic
  childNeeds: { fullness: number; hygiene: number; social: number; inspiration: number };
  transferRate: number;                 // Per-tick rate, derived from parent's Pregnancy expression
  gestationStartTick: number;

  start(opts: {
    childDna: string;
    childFamilyName: string;
    childFactionId: string | null;
    partnerId: string | null;
    transferRate: number;
    startTick: number;
  }): void;

  /**
   * Per-tick transfer step. Returns the amount drained from each of the parent's needs
   * (the parent's needs decay by this amount on top of their normal decay).
   */
  tickTransfer(): { fullnessDrained: number; hygieneDrained: number; socialDrained: number; inspirationDrained: number };

  isReadyForBirth(): boolean;            // True when all child needs >= TUNE.pregnancy.completionThreshold

  end(): void;
}
```

### 5.2 Reproduce Effects (`action/effects/reproduce-effects.ts`)

```typescript
export function applyReproduce(
  parent: Agent,
  partner: Agent | null,
  factory: AgentFactory,
  bus: EventBus,
  currentTick: number
): void {
  // Determine mutation rate from Volatility
  const rate = partner
    ? (parent.traits.volatility.mutationRate + partner.traits.volatility.mutationRate) / 2
    : parent.traits.volatility.mutationRate;

  // Generate child DNA
  const childDna = partner
    ? mutate(crossover(parent.genome.dna, partner.genome.dna), rate)
    : mutate(parent.genome.dna, rate);

  // Check pregnancy expression — does the parent gestate?
  if (parent.traits.pregnancy.gestationMs > 0) {
    // Start pregnancy state — child is NOT born yet
    parent.pregnancy.start({
      childDna,
      childFamilyName: deriveFamilyName(parent, partner),
      childFactionId: parent.factionId,
      partnerId: partner?.id ?? null,
      transferRate: TUNE.pregnancy.needTransferRate,
      startTick: currentTick,
    });
    bus.emit('pregnancy:started', { agentId: parent.id });
  } else {
    // Instant birth — no gestation (matches v4 behavior)
    const child = factory.createChildFromDna(childDna, parent, partner, parent.cellX, parent.cellY);
    if (child) {
      bus.emit('agent:born', { childId: child.id, parentIds: [parent.id, partner?.id].filter(Boolean) });
    } else {
      bus.emit('agent:stillborn', { parentId: parent.id });
    }
  }
}
```

The pregnancy transfer logic runs in `agent-updater.ts` per tick:

```typescript
function processPregnancyTransfer(agent: Agent, currentTick: number, bus: EventBus): void {
  if (!agent.pregnancy?.active) return;

  const drained = agent.pregnancy.tickTransfer();
  agent.drainFullness(drained.fullnessDrained);
  agent.needs.hygiene = Math.max(0, agent.needs.hygiene - drained.hygieneDrained);
  agent.needs.social = Math.max(0, agent.needs.social - drained.socialDrained);
  agent.needs.inspiration = Math.max(0, agent.needs.inspiration - drained.inspirationDrained);

  if (agent.pregnancy.isReadyForBirth()) {
    const child = createChildFromPregnancy(agent);
    if (child) {
      bus.emit('agent:born', { childId: child.id, parentIds: [agent.id] });
    }
    agent.pregnancy.end();
  }
}
```

## 7. World Generator (`world/world-generator.ts`) — NEW

Replaces v4's inline world setup logic in `main.ts`.

```typescript
export class WorldGenerator {
  generate(opts: { gridSize: number; obstacleDensity: number; pocketCount: number }): GeneratedWorld {
    // 1. Generate base terrain (noise-based)
    const terrain = this.generateTerrain(opts.gridSize);

    // 2. Place water bodies
    const water = this.placeWater(terrain);

    // 3. Place obstacle barriers to create isolation pockets
    const obstacles = this.createIsolationBarriers(terrain, opts.pocketCount);

    return { terrain, water, obstacles };
  }

  private createIsolationBarriers(terrain: TerrainGrid, pocketCount: number): IPosition[] {
    // Voronoi-style: pick N random seeds, assign each cell to nearest seed,
    // place obstacles along boundaries between regions
    const seeds = pickRandomSeeds(pocketCount, terrain.width, terrain.height);
    const regions = assignRegions(terrain, seeds);
    return findRegionBoundaries(regions);
  }
}
```

The exact algorithm is tunable; the principle is "carve the map into N connected regions with obstacle barriers between them." Existing pathfinding and grid systems are unchanged.

## 8. Rendering Additions

### 8.1 Animation Runner (`rendering/animation-runner.ts`) — NEW

Tracks per-agent animation state. Each animation is a short transformation (translation, rotation, scale) applied during rendering.

```typescript
export interface AnimationState {
  type: AnimationType;
  startTime: number;
  durationMs: number;
}

export interface AnimationTransform {
  dx: number;
  dy: number;
  rotation: number;
  scale: number;
}

export type AnimationType =
  | 'shake' | 'bob' | 'pulse' | 'bounce'
  | 'shrink' | 'wiggle' | 'flash' | 'sway' | 'none';

export class AnimationRunner {
  private states = new Map<string, AnimationState>();   // agentId → state

  trigger(agentId: string, type: AnimationType, durationMs: number = 500): void {
    this.states.set(agentId, { type, startTime: performance.now(), durationMs });
  }

  getTransform(agent: Agent): AnimationTransform {
    const stateAnim = this.states.get(agent.id);
    const actionAnim = agent.action ? ACTION_ANIMATIONS[agent.action.type] : 'none';
    const animType = stateAnim?.type ?? actionAnim;
    return this.computeTransform(animType, stateAnim?.startTime ?? performance.now(), stateAnim?.durationMs ?? 500);
  }

  private computeTransform(type: AnimationType, startTime: number, duration: number): AnimationTransform {
    const elapsed = performance.now() - startTime;
    const t = Math.min(1, elapsed / duration);

    switch (type) {
      case 'shake':
        return { dx: Math.sin(elapsed / 30) * 3 * (1 - t), dy: 0, rotation: 0, scale: 1 };
      case 'bob':
        return { dx: 0, dy: Math.sin(elapsed / 200) * 2, rotation: 0, scale: 1 };
      case 'pulse':
        return { dx: 0, dy: 0, rotation: 0, scale: 1 + Math.sin(elapsed / 200) * 0.1 };
      case 'bounce':
        return { dx: 0, dy: -Math.abs(Math.sin(elapsed / 150)) * 6 * (1 - t), rotation: 0, scale: 1 };
      case 'shrink':
        return { dx: 0, dy: 0, rotation: 0, scale: 1 - t };
      case 'wiggle':
        return { dx: 0, dy: 0, rotation: Math.sin(elapsed / 80) * 0.1, scale: 1 };
      case 'flash':
        return { dx: 0, dy: 0, rotation: 0, scale: 1 + Math.sin(elapsed / 100) * 0.2 };
      case 'sway':
        return { dx: Math.sin(elapsed / 1000) * 1, dy: 0, rotation: Math.sin(elapsed / 1000) * 0.05, scale: 1 };
      case 'none':
      default:
        return { dx: 0, dy: 0, rotation: 0, scale: 1 };
    }
  }
}
```

### 8.3 Indicator Renderer (`rendering/indicator-renderer.ts`) — NEW

```typescript
export interface IndicatorSlotConfig {
  source: 'faction_flag' | 'pregnancy' | 'health_band' | 'mood' | 'level' | 'none';
}

export class IndicatorRenderer {
  constructor(private slotConfig: {
    topLeft: IndicatorSlotConfig;
    topRight: IndicatorSlotConfig;
    topMiddle: IndicatorSlotConfig;
  }) {}

  render(ctx: CanvasRenderingContext2D, agent: Agent, sx: number, sy: number, scale: number): void {
    const indicatorSize = CELL_PX * scale * 0.5;

    const positions = {
      topLeft:   { sx: sx - CELL_PX * scale * 0.3, sy: sy - CELL_PX * scale * 0.4 },
      topRight:  { sx: sx + CELL_PX * scale * 0.3, sy: sy - CELL_PX * scale * 0.4 },
      topMiddle: { sx: sx,                          sy: sy - CELL_PX * scale * 0.5 },
    };

    this.renderSlot(ctx, agent, this.slotConfig.topLeft, positions.topLeft, indicatorSize);
    this.renderSlot(ctx, agent, this.slotConfig.topRight, positions.topRight, indicatorSize);
    this.renderSlot(ctx, agent, this.slotConfig.topMiddle, positions.topMiddle, indicatorSize);
  }

  private resolveIndicatorEmoji(agent: Agent, source: string): string | null {
    switch (source) {
      case 'faction_flag':
        return agent.factionId ? this.world.factions[agent.factionId].flag : null;
      case 'pregnancy':
        return agent.pregnancy?.active ? '🥚' : null;
      case 'health_band': {
        const ratio = agent.health / agent.maxHealth;
        if (ratio > 0.7) return '💚';
        if (ratio > 0.3) return '💛';
        return '❤️';
      }
      case 'mood':
        return agent.currentMoodEmoji ?? null;
      case 'level':
        return null; // Level uses text rendering, not emoji
      case 'none':
        return null;
    }
  }
}
```

### 8.4 Tool Renderer (`rendering/tool-renderer.ts`) — NEW

Preserves v4's directional tool rendering, generalized to all actions with `targetType === 'external_cell'`.

```typescript
export class ToolRenderer {
  render(ctx: CanvasRenderingContext2D, agent: Agent, action: IActionState, camera: Camera): void {
    const def = ACTION_REGISTRY.get(action.type);
    if (!def?.tool || def.targetType !== 'external_cell') return;
    if (!action.target) return;

    const targetPos = this.resolveTargetPosition(action.target);
    if (!targetPos) return;

    const { sx: ox, sy: oy } = camera.worldToScreen(agent.cellX, agent.cellY);
    const { sx: tx, sy: ty } = camera.worldToScreen(targetPos.x, targetPos.y);

    const mx = (ox + tx) / 2;
    const my = (oy + ty) / 2;
    const angle = Math.atan2(ty - oy, tx - ox);

    ctx.save();
    ctx.translate(mx, my);
    ctx.rotate(angle);
    this.emojiCache.drawAt(ctx, def.tool, 0, 0, CELL_PX * camera.scale * 0.6);
    ctx.restore();
  }
}
```

## 11. Implementation Phases

Each phase produces a working build. Phases are ordered by dependency.

**Phase 1: Foundation**
- New `core/tuning.ts`
- New `genetics/cost-functions.ts`
- Update `genetics/expression.ts` to remove hard caps
- Update `genetics/viability.ts` (minor — same checks, no new ones)
- Update `genetics/mutation.ts` to take rate parameter
- Add AD, AG, AP to `genetics/gene-registry.ts`
- Update `genetics/types.ts` with new TraitSet fields

## 12. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| **Trait ceiling removal causes runaway stats** | Agents become unkillable or omnipotent | Cost functions must scale aggressively enough; tune TUNE.cost coefficients early |
| **Volatility default too high** | Mass stillbirths → extinction | Start with default matching v4's 0.5% rate; only high-Volatility lineages take extra risk |
| **Pregnancy transfer drains parents to death** | Pregnant agents always die | Tune transfer rate and completion threshold; add a floor so transfer pauses when parent needs are critical |
| **Geographic isolation too strong** | Populations can't recover from collapse in small pockets | Ensure barriers have occasional gaps; tune pocket count and density |
| **OO→AD migration breaks existing saves** | Agents lose social behavior on load | Gene registry accepts both OO and AD during expression; migration is transparent |
| **Animation overhead for 300 agents** | FPS drop | AnimationRunner uses simple math (sin/cos), no allocations per frame; profile early |

## 13. Open Technical Questions

### 13.1 Cost Scaling Curves
`cost-functions.ts` uses simple linear scaling. Some traits may need quadratic scaling to feel right (e.g., very high Strength should be disproportionately expensive). Defer to Phase 1 playtest.

### 13.2 Memoized Cost Values
Should `passiveEnergyDrainPerTick`, `attackEnergyCost`, etc. be precomputed on the Agent at creation (since traits are immutable)? Suggested: yes, add cached fields. Implement during Phase 1.

### 13.3 World Generator Algorithm
Voronoi-based region splitting vs noise-based vs maze carving. Voronoi is simplest and produces clear regions. Decide during Phase 5.

### 13.4 Pregnancy Need Transfer Rate Formula
The transfer rate is currently flat (`TUNE.pregnancy.needTransferRate`). Should it scale inversely with `Pregnancy.gestationMs`? Longer gestation = slower transfer per tick = less per-tick burden but longer total burden. Suggested: `transferRate = TUNE.pregnancy.completionThreshold / gestationMs * tickMs`.

### 13.5 Animation State Storage
`AnimationRunner` stores per-agent state in a Map. Alternative: store animation state directly on the Agent. Map decouples rendering from entity; field-on-agent is faster. Decide during Phase 4.
